# 安全测试文档分析技能

## 技能定位

这个技能用于**分析资料文档**（不是源代码），从**安全测试视角**识别出系统中的高危业务功能，并给出针对性的安全测试建议。

### 核心理念

文档描述了系统"要做什么"——从中可以识别出哪些业务功能一旦出现安全问题影响最大（数据泄露、网络中断、权限失控等），从而指导安全测试的优先级。这是"左移"的安全测试分析方法，在开发实施前就能识别风险。

### 与代码分析的区别

| 本技能 | 非本技能范围 |
|--------|-------------|
| 分析需求文档、设计文档、操作指南、接口规范 | 分析源代码（.py/.java/.js 等） |
| 识别高危业务功能 | 找代码漏洞（SQL注入、XSS等） |
| 从业务角度评估风险 | 做代码静态分析 |
| 给出业务层面的安全测试建议 | 给出代码修复建议 |

## 环境准备与依赖安装

本技能处理 CHM 文件时需要安装依赖：

```bash
pip install -r requirements.txt
```

当前依赖：

| 包名 | 用途 | 安装命令 |
|------|------|---------|
| python-chm | CHM 文件解析与内容提取 | `pip install python-chm` |

如果无法安装 `python-chm`，也可使用系统自带的 `extract_chmLib` 命令行工具替代 CHM 提取功能。

## 工作流程

采用**并行分析 + 汇总**的模式，提高处理大量文档的效率和效果。

### 阶段一：文档发现与分组

1. **发现文档**：使用 Glob 工具在用户指定的目录中查找文档文件
   - 支持格式：`*.docx`, `*.doc`, `*.pdf`, `*.chm`, `*.md`, `*.txt`
   - 注意记录文件路径和格式

2. **文档分组**：将文档按主题或关联性分组。每组一个或多个相关文档成为后续一个独立子任务的分析对象。
   - 示例：将"需求规格说明书"分为一组、"操作指南"分为一组、"接口规范"分为一组
   - 如果文档很少（1-2个），可以一组处理所有文档

### 阶段一附加：文档大小评估与 CHM 强制处理路径

CHM 文件无论大小，都必须先解压并解析出章节目录路径，走 CHM 处理流水线。其他格式按大小选择路径：

| 文档类型 | 策略 |
|---------|------|
| **CHM 格式** | **强制走 CHM 处理流水线（阶段二备选）**，先解压再解析全量章节目录路径 |
| 非 CHM 格式，所有文件 < 1MB（预估文本 < 200KB） | 走标准**阶段二**（现有流程） |
| 非 CHM 格式，有文件 ≥ 1MB | 走**大文件流水线（阶段二备选）** |

- 对 PDF/DOCX：检查文件大小做初步判断
- 如果分组内混有 CHM 文件，CHM 文件走流水线，其余文件走标准流程

### 阶段二：并行子任务分析（标准流程，适用于中小文件）

3. **启动子任务**：对每个文档分组，启动一个独立的 Agent 子任务并行分析。

每个子任务需要执行以下操作：

#### 3.1 读取参考文件
在分析前，先读取以下文件获取分析框架：
- `references/high-risk-functions.md` — 高危业务功能识别框架

#### 3.2 提取文档内容
根据文档格式采用不同方式提取文本内容：
- **docx 格式**：使用 python-docx 库（`python3 -c "import docx; doc=docx.Document('path'); print('\n'.join([p.text for p in doc.paragraphs]))"`）
- **doc 格式**：使用 antiword 或 catdoc 命令行工具
- **pdf 格式**：使用 pdfplumber 或 pdftotext 提取文本
- **md/txt 格式**：直接使用 Read 工具读取

> CHM 文件不经过此标准流程，统一走 CHM 处理流水线（阶段二备选）。

> 如果文档提取失败（扫描件、加密文件等），记录下来并跳过，在报告中说明。

#### 3.3 识别业务功能
通读提取的文档内容，列出系统中有哪些业务功能。注意区分：
- **核心业务功能**：支撑系统主要业务目标的（如配置管理、设备管理等）
- **辅助功能**：次要的系统功能（如个人设置、主题切换等）

#### 3.4 高危业务研判
对照 `references/high-risk-functions.md` 中的识别模式，逐功能评估是否属于高危。核心识别三类高风险业务：
- **涉及文件或路径的**：上传、下载、导入、导出、备份路径配置、SFTP/FTP配置等
- **涉及与第三方对接的**：北向接口、webhook、数据同步、HTTPS证书配置等
- **涉及配置密码或凭据的**（不含登录本身）：SFTP密码、API Token、SNMP Community、共享密钥等

同时记录非高危功能作为对比（如信息查看、统计报表、个人设置等）。

#### 3.5 分析业务影响
对每个高危功能，思考如果被攻击或滥用会造成什么后果：
- 数据泄露：什么数据可能泄露？影响多大？
- 业务中断：什么业务会中断？影响范围？
- 权限失控：攻击者能获得什么权限？会影响哪些系统？
- 合规风险：违反等保、网络安全法等法规？

#### 3.6 生成安全测试建议
针对每个高危功能，给出具体可操作的安全测试建议，包含：
- **测试类型**：功能安全验证 / 异常场景测试 / 权限测试 / 传输安全测试
- **测试步骤**：具体怎么测
- **预期结果**：安全的系统应该表现为什么行为
- **检查点**：测试时需要关注什么

#### 3.7 输出中间分析结果
每个子任务输出结构化分析结果（markdown 格式），包含：
- 文档名称和分组信息
- 识别出的业务功能列表（含高危标记）
- 每个高危功能的详细分析
- 安全测试建议

### 阶段二备选：CHM 处理流水线（CHM 文件专用，强制使用）

CHM 文件必须使用本流水线，不经过标准阶段二。核心流程：**先解压 CHM、提取目录结构（TOC）做全量语义分析，识别出可疑章节后再逐章深度分析**。非 CHM 的大文件（≥1MB）也可使用本流水线的高效处理逻辑。

#### 阶段二-A：解压并提取 TOC 目录结构

**CHM 文件**：
1. 使用 CHM 提取工具的 `--toc` 模式提取目录结构（例如：`python3 scripts/extract_chm.py --toc "<chm_path>"`）
2. 解析输出的 JSON，获取章节列表（含标题、路径、层级、parent_id、full_path）
3. 确认 TOC 覆盖了文档的所有章节，无遗漏

TOC 结构示例：
```json
[
  {"id": 1, "title": "第一章 设备管理",  "path": "/ch1.htm",         "level": 1, "parent_id": null},
  {"id": 2, "title": "1.1 上传软件包",    "path": "/ch1/sec1.htm",    "level": 2, "parent_id": 1},
  {"id": 3, "title": "1.2 备份配置",      "path": "/ch1/sec2.htm",    "level": 2, "parent_id": 1},
  {"id": 4, "title": "1.2.1 SFTP设置",    "path": "/ch1/sec2_1.htm",  "level": 3, "parent_id": 3}
]
```

**对其他格式**（非 CHM）：按标题层级提取目录结构，包括 DOCX 的 Heading 样式、PDF 的章节标题、MD 的 `#` 标记。

#### 阶段二-B：TOC 全量语义分析——仅根据章节标题初步研判（核心步骤）

**注意：此阶段只根据章节标题和路径名进行研判，不读取任何章节的具体内容。** 将完整的目录 JSON 一次性交给 LLM 做全量语义分析。**这是与关键词匹配的关键区别**：利用 LLM 理解章节标题的业务语义，识别疑似高危章节。

**为什么仅凭章节标题就足够做初筛：**
- 目录标题本身就是文档的索引，"SFTP配置"、"设备软件包上传"、"北向接口对接"直接反映业务功能
- 层级关系提供上下文：父章节"参数配置"下含"SFTP密码"、"Token设置"子章节，能推断为凭据配置相关
- 目录数据量很小（几百章节通常 < 50KB），完全能放入 LLM 上下文做全量分析

**分析要求**：

1. **逐章节研判**：通读所有章节标题（含父级路径 `full_path`），对 TOC 中每一个节点做研判
2. **识别三类高危模式**：文件/路径操作、第三方对接、凭据配置
3. **标记"待确认"**：标题模糊无法判断的（如"其他功能"、"高级设置"），标记为"待确认"——不会漏掉，后续阶段会验证
4. **层级穿透**：如果父章节高危，自动标记所有子章节为"需关注"；如果父章节看似安全但多个子章节高危，标记父章节为"含高危子章节"
5. **关注复合风险**：同时命中多个模式的（如 SFTP 既涉及文件传输又涉及密码配置），标记为高优先级

**输出结构化结果**（JSON），每个章节标记一个风险类别：
```json
[
  {"id": 2,  "title": "1.1 上传软件包",    "risk": "高危",   "reason": "涉及文件上传操作",                 "patterns": ["文件/路径"]},
  {"id": 4,  "title": "1.2.1 SFTP设置",    "risk": "高危",   "reason": "涉及文件传输和密码配置",           "patterns": ["文件/路径", "凭据配置"]},
  {"id": 5,  "title": "第一章 系统概述",    "risk": "低危",   "reason": "系统介绍，不涉及具体业务功能"},
  {"id": 10, "title": "3.2 高级设置",       "risk": "待确认", "reason": "标题模糊，需检查具体内容"}
]
```

将此结果保存为 `_toc_analysis.json`（见下方解压目录约定）。

#### 阶段二-C：依据研判结果决定是否读取内容

**核心原则：只读取 TOC 研判为"高危"或"待确认"的章节具体内容，"低危"章节不读内容，直接跳过。** 这样既保证不遗漏，又只在必要时才消耗 tokens 读取详细内容。

| TOC 研判结果 | 处理方式 | 是否读取章节内容 |
|-------------|---------|----------------|
| **高危** | 直接进入深度分析（阶段二-D），读取全量章节内容后执行完整分析 | **是** |
| **待确认** | 先读取章节开头片段（~1500 字符）做快速验证，确认为高危则进入深度分析，否则标记为低危跳过 | **仅读开头片段** |
| **低危** | 跳过，不读取任何内容，在报告中记录为"经 TOC 研判为低危" | **否** |

**待确认章节的验证步骤**：
1. 启动独立子任务处理该章节：提取开头片段（`python3 scripts/extract_chm.py --section "<path>" "<chm_path>" | head -c 1500`）并判断是否涉及高危业务
2. 确认为高危的，标记为"高危"并进入阶段二-D 深度分析
3. 确认为低危的，更新 `_toc_analysis.json` 中的标记

#### 阶段二-D：为每个高危章节启动独立子任务深度分析

对阶段二-B 和阶段二-C 最终标记为"高危"的每个章节，**分别为每个章节启动一个独立的 Agent 子任务**并行分析。每个子任务只负责分析分配给它的单个章节，互相独立，输出各自的结构化分析结果：

1. **全量提取章节内容**：
   - CHM 文件：使用 CHM 提取工具的 `--section` 模式提取（例如：`python3 scripts/extract_chm.py --section "<path>" "<chm_path>"`）
   - 其他格式：提取文档的对应部分

2. **摘要级联**（如果单节内容超过 ~8K tokens）：
   a. 按段落或子标题将章节拆分为 ~6K tokens 的块
   b. 对每个块生成摘要（重点：业务功能描述、高危模式命中情况、凭据/协议等细节）
   c. 合并所有块的摘要
   d. 对合并后的摘要执行标准分析（识别业务功能 → 高危研判 → 业务影响 → 测试建议）
   e. 如果摘要指示某个块包含高风险内容，定向回读该块原文做精确分析

3. **执行标准分析**（同 3.3-3.6）：
   - 读取 `references/high-risk-functions.md`
   - 识别业务功能
   - 高危研判
   - 业务影响分析
   - 安全测试建议

#### 阶段二-E：组内合并

所有章节子任务完成后：

1. 合并各组章节层面的分析结果
2. 去重合并跨章节的同一业务功能
3. 记录 TOC 全量分析的研判结果（每章节的归类）到最终报告
4. 输出统一的组级分析结果

### 统一解压目录约定

所有 CHM 文件必须统一解压到以下目录结构存放中间产物：

```
工作目录/
└── .tmp_chm_extract/
    └── <CHM文件名（去扩展名）>/
        ├── _toc.json              # 目录结构（阶段二-A 输出）
        ├── _toc_analysis.json     # TOC 语义分析结果（阶段二-B 输出）
        ├── ch001_章节标题.txt      # 各章节解压后的纯文本
        ├── ch002_章节标题.txt
        └── ...
```

**命名规则**：
- 顶层：`.tmp_chm_extract/`（隐藏目录，表示中间产物）
- 二级：以 CHM 文件名去扩展名命名，多个 CHM 文件互不冲突
- 章节文件：`ch<序号>_<标题>.txt`，序号按 TOC id 编排
- 元数据文件：`_toc.json`、`_toc_analysis.json`，加下划线前缀区别于内容文件

**批量解压命令**：
```bash
# 使用 CHM 提取工具的 --extract-dir 模式
python3 scripts/extract_chm.py --extract-dir ".tmp_chm_extract/<文档名>" "<chm文件路径>"
```

**清理**：报告生成完成后自动删除解压目录；异常中断时保留目录便于排查。

### 阶段三：汇总与报告生成

4. **等待所有子任务完成**

5. **汇总分析结果**：
   - 合并所有子任务的分析结果
   - 去重合并（同一业务功能可能出现在多个文档中）
   - 统一风险等级标准，确保跨组一致性

6. **按报告模板生成最终报告**：
   - 读取 `references/report-template.md` 获取报告模板
   - 填入分析内容
   - 生成完整的 markdown 报告文件

- 报告格式为 Markdown
- 报告内容应全面覆盖所有分析结果

## 关键设计约束

1. **基于文档内容分析**：只分析文档中描述的内容，不要猜测未文档化的实现
2. **文档提取失败处理**：遇到扫描件 PDF、加密文件等无法提取时，记录告警并跳过
3. **分析局限性说明**：文档信息不足时，在报告中说明分析的局限性
4. **风险等级区分**：避免把所有功能都标为严重，要合理区分风险等级
5. **不做代码级分析**：不要映射 OWASP 漏洞类型，不要做代码级安全分析
6. **建议要具体**：安全测试建议要具体可操作，不要泛泛而谈
7. **CHM 文件强制流程**：所有 CHM 文件必须走 CHM 处理流水线（阶段二备选），不得全量提取后单次分析。必须先解压到 `.tmp_chm_extract/<文档名>/` 目录，再解析出全量章节目录路径（`full_path`），通过 TOC 全量语义分析保证不遗漏疑似高危章节，最后对可疑章节逐章深度分析。
